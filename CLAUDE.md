# Project: Claude Router

## Project structure

This is a **Wails3** desktop application.

- `frontend/` — Vue frontend (TypeScript). All frontend rules below apply to code inside this folder.
  - `frontend/bindings/` — auto-generated by Wails3. **Never edit manually** — regenerate via Wails CLI when the Go API changes.
- `core/` — Go backend code (business logic, services, Wails bindings).

---

> **Scope note:** Everything below this line applies exclusively to the `frontend/` folder.

---

## Architecture — layered (Clean Architecture, hybrid)

The frontend is organised into **layers** at the top level, with feature subfolders inside each. Dependencies point **inwards only**: `presentation → application → domain`, and `application → infrastructure → domain`. `domain` depends on nothing (no Vue, no tsyringe runtime, no Wails bindings, no IO).

```
src/
  domain/                      # entities, value-objects, domain events, pure helpers — NO IO, NO Vue
    models/                    # one class/type per file; barrels (index.ts) keep historic import paths
      instance/                # InstanceModel (isolated Claude Desktop instance metadata — no runtime status)
    events/                    # one event class per file, grouped by area
      instances/                # InstanceCreatedEvent, InstanceRemovedEvent, InstanceRenamedEvent, InstancesChangedEvent, InstanceLaunchRequestedEvent, InstanceStopRequestedEvent, InstanceOpenFolderRequestedEvent, InstanceFolderDeletionRequestedEvent, InstancesPersistFailedEvent
      settings/                 # SettingsChangedEvent, SettingsPersistFailedEvent
    support/                   # framework-free static helpers: EntityRegistry, Slugifier
    decorators/                # @Entity decorator
  application/
    services/                  # use-case services (InstanceService, InstanceProcessService, IdService)
    handlers/                  # event subscribers that perform side-effects (see “Event system”)
    di/                        # container.ts (auto-registration) + reflect-lite
  infrastructure/              # adapters over Wails bindings / IO / persistence
    repositories/              # InstanceConfigRepository, InstanceProcessRepository, SettingsRepository (wrap the generated Go bindings)
    wails/                     # WailsRuntimeService (runtime/bindings availability)
  services/eventBus/           # EventBus (mitt-backed)
  store/ views/ components/    # presentation (Pinia stores, pages, Vue components)
  lib/                         # framework infra: router, vue-store, extendedTypes, utils
```

### Core principle — event-driven side-effects

**A state change emits an event; the side-effect happens in a handler elsewhere — never inline in the store.** Stores mutate their own reactive state (via pure `domain`/`application` transforms) and **emit events**; they must not call persistence, routing-sync, docker, or other side-effecting services directly. Subscribers in `application/handlers/` react to those events and perform the work, then report results back via events or narrow store setters. See **Event system — rules**.

### No mock data

Mock/fake data must not live in runtime code. Real domain catalogues (e.g. the managed processes) belong in `domain/` as genuine models (`ProcessCatalog`) with no invented stats; runtime values come from live sources. Test fixtures stay in `src/test/`.

## General conventions

### Naming

| Artifact | Convention | Example |
|---|---|---|
| Class | `PascalCase` | `TrainingProgramStore` |
| Model | `PascalCase` | `SessionSummary` |
| Type | `T` prefix + `PascalCase` | `TUserGender`, `TFilterOptions` |
| Interface | `I` prefix + `PascalCase` | `IDropHandler`, `IPasteHandler` |
| Vue component file | `PascalCase.vue` | `WorkoutRunner.vue` |
| Vue component in template | `kebab-case` | `<workout-runner />` |

These conventions apply to everything under `src/` **except**:
- `src/views/` — keep existing file/folder naming as-is (kebab-case folders, `index.vue` pages, `*.route.ts` route files).
- `src/` root-level files — keep as-is.

### Interfaces vs types

`interface` is an OOP contract — it describes **behaviour** that a class will implement (methods, not just fields):

```ts
// correct — describes behaviour
export interface IDropHandler {
    canHandle(event: DragEvent): boolean
    handle(event: DragEvent, editor: ICodeEditor): Promise<void>
}

export class ImageDropHandler implements IDropHandler { ... }
```

`type` is a data shape — use it for everything else: plain data structures, union types, mapped types, aliases.

```ts
// correct — data shape
export type TUserGender = 'male' | 'female'
export type TFilterOptions = { field: string; value: unknown }

// wrong — interface used as a plain data shape, use type instead
export interface UserGender { value: string }
```

**Never use `interface` where `type` would do** — i.e. when the interface only describes fields and no class will `implement` it.

### No standalone functions — OOP only

**`export function` is forbidden.** All logic must live inside a class as a method.

- ✅ `class StringUtils { static truncate(...) {} }` — static method on a class
- ✅ `class FormatterService { format(...) {} }` — instance method on a service
- ❌ `export function truncate(...)` — bare exported function
- ❌ `export const helper = () => ...` — exported arrow function at module level

This applies everywhere: services, stores, utilities, helpers, composables. If you feel the urge to write a standalone function, put it as a `static` method on a utility class or as an instance method on a service.

### One artifact per file

- **One class per file.** Never define two classes in the same `.ts` file.
- **One type per file.** Each type or interface gets its own `T{Name}.ts` file — except types that are tightly coupled to each other and have no meaning in isolation (e.g. a pair of request/response types for the same operation), which may share a file.
- **One component per file.** Each Vue component lives in its own `.vue` file.

## Domain layer — rules (`src/domain/`)

The domain layer holds entities, value objects, domain events and pure helpers. It is **framework-free**: no Vue, no `tsyringe` runtime, no Wails bindings, no IO. Other layers import from it; it imports from nothing outside `domain/`.

### Models (`src/domain/models/`)

Each model is an immutable class in its own file. Models are plain classes (they do **not** extend a base repo class) and are decorated with `@Entity` for metadata:

```ts
import { Entity } from '@/domain/decorators/entity-metadata'
import type { ProviderType } from '@/domain/models/application/types/ProviderType'

export type ProviderProps = {
    id: string
    name: string
    type: ProviderType
    active: boolean
}

@Entity({ name: 'Provider', collection: 'providers', displayName: 'Provider' })
export class ProviderModel implements ProviderProps {
    readonly id: string
    readonly name: string
    readonly type: ProviderType
    readonly active: boolean

    constructor(props: ProviderProps) { Object.assign(this, props) }
    static create(props: ProviderProps): ProviderModel { return new ProviderModel(props) }
    withChanges(changes: Partial<ProviderProps>): ProviderModel { return new ProviderModel({ ...this, ...changes }) }
}
```

Rules:

1. **One class per file**, file name matches the class name exactly (`ProviderModel.ts`).
2. **Props as `type`, co-located** — a model's `{Name}Props` data shape is a `type` (never `interface`) and lives in the same file as the model that `implements` it.
3. **Shared scalar/union types** (`ProcessStatus`, `ProviderType`, `LogLevel`, …) each get their own file under the feature's `types/` subfolder.
4. **Immutability** — fields are `readonly`; mutation is expressed via `withChanges(...)`/derived helpers that return a **new** instance. No invented placeholder values (e.g. `start()` must not fabricate fake cpu/uptime).
5. **`@Entity` decorator** — from `@/domain/decorators/entity-metadata`; metadata is stored in `EntityRegistry` (`@/domain/support/EntityRegistry`).
6. **Imports** — always via the `@/` alias, never relative paths.

### Barrels preserve historic import paths

Each feature folder has an `index.ts` barrel re-exporting its classes/types, so the folder resolves at its historic path (e.g. `@/domain/models/instance`). Add new files to the matching barrel. This keeps “one class per file” without churning every importer.

### Domain events (`src/domain/events/`)

One event class per file, grouped by area: `instances/`, `settings/` (extend with new areas as new domains are added). Plain classes, fields as `public readonly` constructor params (primitives preferred; entities allowed when the handler needs the full object). See **Event system — rules**.

### Support helpers (`src/domain/support/`)

Framework-free reusable logic as **static methods on a class** (e.g. `EntityRegistry`). Never module-level functions.

## Vue Components — rules (`src/components/`, `src/views/`, `src/domain/`)

### Key principle

**All components are classes.** The project uses a class-based component style via `@i-app/vue-facing-di`. Composition API (`<script setup>`, `defineComponent`, `ref`, `reactive`, etc.) is **never used**.

### ⚠️ Extract early — components are cheap, bloated templates are not

**Any distinct piece of UI functionality must live in its own component.** Do not wait for a template to become "too big" — extract as soon as a block has its own purpose.

Rules of thumb for when to extract:

- The block has its own state, loading flag, or lifecycle logic.
- The block could be described with a noun: `PostCard`, `CommentForm`, `PriceTag`, `EmptyState`.
- The block repeats inside a `v-for` (even once — extract it).
- The block is conditionally shown (`v-if`) and has non-trivial content.
- The parent template exceeds ~30 lines of meaningful markup.

**Err on the side of too many components.** A component that turns out to be used only once is still the right call — it keeps each file focused and makes future reuse trivially easy. Never inline a block "because it's only used here."

```html
<!-- wrong — inline block with its own visual purpose -->
<div class="user-card">
  <img :src="user.avatar" />
  <span>{{ user.name }}</span>
  <badge :type="user.role" />
</div>

<!-- correct — extracted to its own component -->
<user-card :user="user" />
```

### File structure

A single `.vue` file with `<template>` and `<script lang="ts">`.

```vue
<template>
  <!-- ... -->
</template>

<script lang="ts">
import { Component, VueBase } from '@i-app/vue-facing-di'

@Component({})
export default class MyComponent extends VueBase {
  // ...
}
</script>
```

### `@Component` decorator

Required on every class. Options used:

| Option | Purpose |
|---|---|
| `components` | Register child components |
| `emits` | Declare emitted events |
| `directives` | Register custom directives |

```ts
@Component({
  components: { SomeChild, SomeIcon },
  emits: ['close', 'submit'],
  directives: { scroll: vScroll },
})
```

### Props

Use `@Prop()` from `@i-app/vue-facing-di`. Mark as `public readonly`.

```ts
@Prop({ required: true })
public readonly title: string

@Prop({ required: false })
public readonly type?: string

@Prop({})
public readonly items: TrainingExercise[]
```

### v-model

Use `@VModel()` from `@i-app/vue-facing-di`:

```ts
@VModel()
public open: boolean
```

### Watchers

Use `@Watch('propertyName')` directly above the handler method:

```ts
@Watch('markdown')
mdChanged(): void {
  this.renderHandler?.()
}
```

### Local reactive state

Plain public class properties (no decorator). TypeScript initializers set the default value.

```ts
public loading = false
public items: SomeType[] = []
private renderHandler: TDebounceHandler<Promise<void>> | null = null
```

- `public` — accessed from the template.
- `private` — internal only.

### Dependency injection

Services are injected in the constructor via `@inject(Token)` from `tsyringe`. Always call `super()`.

```ts
import { inject } from 'tsyringe'
import { SomeService } from '@/services/some/SomeService'

constructor(
  @inject(SomeService) private readonly someService: SomeService,
  @inject(OtherService) public readonly otherService: OtherService, // public when used in template
) {
  super()
}
```

If no services are needed, omit the constructor entirely.

### Lifecycle hooks

Standard Vue hook names as regular class methods. Can be `async`.

```ts
created(): void { ... }
async mounted(): Promise<void> { ... }
beforeUnmount(): void { ... }
```

### Computed properties

TypeScript `get` accessors:

```ts
get exercise(): TrainingExercise {
  return this.exercises[this.exIndex]
}

get progress(): number {
  return (this.done / this.total) * 100
}
```

### Methods

Regular class methods. `public` if called from the template, `private` otherwise.

```ts
public async loadData(): Promise<void> { ... }
private initState(): void { ... }
```

### Emitting events

```ts
this.$emit('close')
this.$emit('submit', payload)
// or inline in template:
// @click="$emit('close')"
```

### Router navigation

```ts
this.$router.push('/some/path')
```

### Imports

- Decorators / base class: `@i-app/vue-facing-di`
- DI: `tsyringe`
- Domain models / events: `@/domain/models/{feature}` (barrel) or `@/domain/events/{area}/{EventName}`
- Services / handlers / infrastructure / stores: `@/application/...`, `@/infrastructure/...`, `@/store/...`, `@/components/...` (always `@/` aliases, never relative paths for cross-folder imports)

### Naming

- Class name and file name — `PascalCase`, must match.
- The `export default` class is the only export from a `.vue` file.

## Pages & Routing — rules (`src/views/`)

### File structure per page

Each page lives in its own folder with exactly two files:

```
src/views/app/program/
├── index.vue            ← page component
└── program.route.ts     ← route class
```

Child pages nest inside the parent folder:

```
src/views/app/templates/
├── index.vue
├── templates.route.ts
└── template-item/
    ├── index.vue
    ├── template-item.route.ts
    └── section/
        ├── index.vue
        └── section.route.ts
```

### Page component (`index.vue`)

Same rules as any Vue component (see section above). Class name ends with `Page`:

```ts
export default class ProgramPage extends VueBase { ... }
```

Route params are accessed via `this.$route.params.paramName as string`. Navigation via `this.$router.push(...)`.

### Route class (`{page-name}.route.ts`)

A plain TypeScript class (no Vue) decorated with router decorators. File named in `kebab-case` matching the page folder name.

```ts
import { RoutePage, RouteTitle } from '@/lib/router/decorators/RouterDecorators'
import { AuthorizedRouteBase } from '@/lib/router/base/AuthorizedRouteBase'
import { AppRoute } from '@/views/app/app.route'

@RoutePage(() => import('./index.vue'), 'program/:programId', () => AppRoute)
@RouteTitle('Программа')
@PropsRoute()
export class ProgramRoute extends AuthorizedRouteBase {
}
```

### Base classes

| Class | When to use |
|---|---|
| `RouteBase` | Public page, no auth check |
| `AuthorizedRouteBase` | Requires login — redirects to `/user/login` if not authorized |
| `AdminRouteBase` | Requires admin role — redirects to `/` otherwise |

Import from `@/lib/router/base/`.

### Route decorators

All imported from `@/lib/router/decorators/RouterDecorators`.

**`@RoutePage(component, path, parent?)`** — required on every route class.
- `component` — lazy import: `() => import('./index.vue')`
- `path` — URL segment relative to parent, e.g. `'dashboard'`, `'program/:programId'`, `':sectionId'`
- `parent` — lazy getter returning parent route class: `() => AppRoute`. Omit for root routes.

**`@RouteRedirect(to)`** — immediately redirect to another path. Used on layout/shell routes that have no own content:
```ts
@RoutePage(() => import('./index.vue'), '/app')
@RouteRedirect('/app/dashboard')
export class AppRoute extends RouteBase {}
```

**`@PropsRoute()`** — passes route params as component props. Add when the page component receives URL params (e.g. `:programId`).

**`@RouteTitle(title)`** — sets a static string title for the page.

**`@RouteIcon(icon)`** — sets the menu icon. Pass a `lucide-vue-next` component directly (not a string):
```ts
import { Layout } from 'lucide-vue-next'
@RouteIcon(Layout)
```

**`@MenuIncluded(order?)`** — includes the route in the sidebar/navigation menu. `order` is a number controlling position.

**`@RouteMeta(data)`** — sets arbitrary metadata (`RouteMetaData`). Prefer the specific decorators (`@RouteTitle`, `@RouteIcon`, etc.) over raw `@RouteMeta`.

**`@AdminRoute()`** — marks route as admin-only in meta (use together with `AdminRouteBase`).

### Dynamic route title

Override `getRouteTitle` when the title depends on a loaded entity:

```ts
export class TemplateItemRoute extends AuthorizedRouteBase {
    async getRouteTitle(route: RouteRecordNormalized, routeParams: Record<string, any>): Promise<string> {
        const store = container.resolve(TemplateEntityStore)
        return store.entities.find(p => p.id.toString() === routeParams.templateId)?.name || ''
    }
}
```

Use `container.resolve(...)` from `tsyringe` to access stores — the route class is not a Vue component and has no DI constructor injection.

## Services — rules (`src/services/`)

### When to create a service

Any logic that is not UI state belongs in a service (or a store). Use a service when:
- The logic does not need to be reactive / shared between components (otherwise use a store).
- It wraps an external library, browser API, or platform capability.
- It is a reusable operation: formatting, transformation, rendering, validation.
- It coordinates multiple other services or performs multi-step domain operations.

### Structure

Plain TypeScript class, decorated with `@injectable()` or `@singleton()` from `tsyringe`. No Vue, no `StoreBase`.

```ts
import { injectable } from 'tsyringe'

@injectable()
export class DeviceService {
    public hapticImpact(style: 'light' | 'medium' | 'heavy' = 'light'): void {
        // ...
    }

    private isNavigatorFuncAvailable(func: string): boolean {
        // ...
    }
}
```

### `@injectable()` vs `@singleton()`

| Decorator | Lifetime | When to use |
|---|---|---|
| `@injectable()` | New instance per resolution | Default. Stateless services, wrappers |
| `@singleton()` | One instance for the whole app | When the service holds its own internal state (e.g. `EventBus` with subscriptions, an initialized engine) |

### Dependency injection

Inject other services in the constructor via `@inject(Token)`:

```ts
import { inject, injectable } from 'tsyringe'
import { OtherService } from '@/services/other/OtherService'

@injectable()
export class MyService {
    constructor(
        @inject(OtherService) private readonly otherService: OtherService,
    ) {}
}
```

No `super()` call needed (services do not extend anything).

> ⚠️ **`@inject(Token)` on every constructor parameter is mandatory.** `src/application/di/reflect-lite.ts` no-ops `Reflect.getMetadata`, so tsyringe cannot infer parameter types from reflection. A bare typed param (without `@inject`) will fail to resolve at runtime. This applies to services, repositories, handlers, and stores alike.

### Registration — no manual wiring

`@injectable()` / `@singleton()` register the class with the container when its module is imported, and `@inject` resolves the graph. **`src/application/di/container.ts` does no manual `registerInstance`/`new`** — just the `reflect-lite` import. Never re-introduce manual wiring; add a decorator instead.

### Layers: application services vs infrastructure

- **`application/services/`** — use-case / domain-coordination services (`InstanceService`, `InstanceProcessService`, `IdService`).
- **`infrastructure/`** — adapters that touch the outside world: `repositories/InstanceConfigRepository`, `repositories/InstanceProcessRepository`, `repositories/SettingsRepository` (wrap the generated Go bindings), `wails/WailsRuntimeService`. Anything reading/writing files or probing the Wails runtime belongs here, behind an injected class — never an inline `"_wails" in window` check or a module-level IO function.

### File & folder layout

```
src/application/services/{name}.service.ts   ← use-case service
src/application/handlers/{Name}Handler.ts    ← event subscriber (side-effects)
src/infrastructure/{repositories,wails}/{Name}.ts
```

- File name matches the class name exactly.
- Helper classes (filters, plugins, providers, actions) live in subfolders within the same service domain.
- Abstract base classes for families of helpers are placed in a `base/` subfolder.

### Public vs private methods

- `public` — part of the service contract, called by consumers.
- `private` — internal implementation detail.

## Stores — rules (`src/store/`)

### Service vs store

| | Service | Store |
|---|---|---|
| State | None (or private/ephemeral) | Reactive, shared across components |
| Decorator | `@injectable()` / `@singleton()` | `@InjectableStore` |
| Base class | none | `StoreBase<Self>` |
| Use when | Operations, transforms, side-effects | Cached API data, UI-global state |

### Plain store

For a single entity or miscellaneous global state extend `StoreBase<Self>` and decorate with `@InjectableStore`:

```ts
import { InjectableStore, StoreBase } from '@/lib/vue-store'
import { inject } from 'tsyringe'
import { ApiContext } from '@/api/ApiContext'
import { User } from '@/api/entities'

@InjectableStore
export class UserStore extends StoreBase<UserStore> {
    public user: User | null = null

    constructor(
        @inject(ApiContext) private readonly apiContext: ApiContext,
    ) {
        super()
    }

    public async loadUser(): Promise<void> {
        if (!this.user) {
            this.user = await this.apiContext.currentUser.query().getOne()
        }
    }
}
```

### Entity-collection store

For stores that cache a paginated or full list of API entities extend `EntitySetStoreBase`:

```ts
import { EntitySetStoreBase } from '@/store/base/EntitySetStoreBase'
import { InjectableStore } from '@/lib/vue-store'
import { inject } from 'tsyringe'
import { ApiContext } from '@/api/ApiContext'
import { TrainingProgram } from '@/api/entities'

@InjectableStore
export class TrainingProgramStore extends EntitySetStoreBase<TrainingProgramStore, TrainingProgram> {
    constructor(
        @inject(ApiContext) private readonly apiContext: ApiContext,
    ) {
        super(apiContext.trainingProgram, /* paginated */ true)
    }
}
```

`EntitySetStoreBase` exposes `entities: TEntity[]`, `loading: boolean`, `load()`, `loadIfNeeded()`, `getAll()`.

### File & folder layout

Every store lives in `src/store/modules/{storeName}/`. The store file is named `{StoreName}Store.ts`. Related files are co-located in typed subfolders inside the same module folder:

```
src/store/modules/userTrainingProgram/
    UserTrainingProgramStore.ts
    models/
        SessionLog.ts
    types/
        TFilterOptions.ts
    utils/
        buildSessionSummary.ts
```

Use subfolders named by type: `models/`, `types/`, `utils/`, `enums/`, `constants/` — whatever applies. Do not mix unrelated store files in the same folder.

### `@InjectableStore`

Replaces both `@injectable()` and Pinia store registration. **Always use it instead of `@injectable()` for stores.** It makes the store a singleton and registers it with the DI container so components can inject it normally.

### Injecting a store in a component

Stores are injected exactly like services — via `@inject(StoreClass)` in the component constructor:

```ts
constructor(
    @inject(UserStore) public readonly userStore: UserStore,
) {
    super()
}
```

Use `public readonly` when the store is accessed from the template.

## Event system — rules

### Key principle

**A state change emits a domain event; the side-effect runs in a handler, not inline.** When a store (or service) changes meaningful state, it emits an event and returns. It must **not** call persistence, routing-sync, docker, process control, or other side-effecting services directly. Dedicated subscribers in `application/handlers/` react to the event and do the work, reporting results back through further events or narrow store setters.

This keeps stores as pure reactive state + emitters, and concentrates orchestration in small, testable handlers.

Within a single component, plain prop/`$emit` and direct method calls are still the right tool for local UI interactions — the event bus is for **domain** state changes and their cross-cutting side-effects, and for cases where more than one place must react.

`EventBus` can be injected anywhere (components, services, stores, handlers). It is registered in the DI container — inject it via constructor like any other service.

### Flow (concrete)

```
InstanceStore.addInstance(name)                  # presentation action
  → InstanceService.addInstance(...)             # pure transform → new instances[]
  → this.instances = next                        # local reactive state
  → emit InstanceCreatedEvent(name)              # NotificationHandler → toast
  → emit InstancesChangedEvent(next)             # InstancePersistenceHandler

InstanceStore.requestLaunch(id)                  # presentation action
  → emit InstanceLaunchRequestedEvent(id, name)  # InstanceLaunchHandler (spawning is not implemented yet)

InstancePersistenceHandler  on InstancesChangedEvent → save; on failure → InstancesPersistFailedEvent
NotificationHandler         on granular instance events → toast
InstanceLaunchHandler       on InstanceLaunchRequestedEvent → will own process spawning/isolation once implemented
```

### Event class

Each event is its own plain class with no decorators. Fields are passed as `public readonly` constructor parameters. Use primitive types for fields; entity types are allowed when the receiver genuinely needs the full object.

```ts
// src/domain/training/events/WorkoutStartedEvent.ts
export class WorkoutStartedEvent {
    constructor(
        public readonly workoutId: number,
        public readonly programId: number,
    ) {}
}

// event with no payload — empty class body
export class WorkoutCompletedEvent {}
```

- One event class per file under `src/domain/events/{area}/` (`applications/`, `routing/`, `docker/`, `processes/`).
- File named `{EventName}.ts`, class named `{EventName}`.
- Every class automatically gets a stable identity (`__type.guid`) — no registration/decorator is needed for the bus to route it.

### Emitting an event

```ts
this.eventBus.emitEvent(new WorkoutStartedEvent(workoutId, programId))
```

### Event handlers (`application/handlers/`)

Cross-cutting side-effects of domain events live in dedicated **handler** classes, not in stores. A handler is a `@singleton()` that injects the `EventBus` (plus whatever services it needs) and registers its subscriptions **in its constructor**:

```ts
@singleton()
export class RoutingSyncHandler {
    constructor(
        @inject(RoutingSyncService) private readonly routingSync: RoutingSyncService,
        @inject(EventBus) private readonly eventBus: EventBus,
    ) {
        this.eventBus.registerHandler(ApplicationsChangedEvent, (e) => this.sync(e))
    }
    // ...
}
```

Handlers are resolved **once** in `AppBootstrap.createApp()` so their subscriptions register exactly one time. A handler may push results back to a store through a narrow setter (e.g. `store.setDockerLoading(...)`, `store.setStorageError(...)`) — that is the allowed handler→store coupling.

### Registering a handler elsewhere

In a **component** — register in `created()`.

> ⚠️ **Do not register bus handlers in a store's `setup()`.** `setup()` runs on every store resolution (once per `useStore()`), so registering there would subscribe duplicates. Register cross-cutting subscriptions in a `@singleton` handler instead. If a store must subscribe (e.g. `ProcessRuntimeStore` reacting to `RoutingSyncedEvent`), do it from a one-shot lifecycle method that runs once (such as `start()`), and unregister in the matching teardown (`dispose()`).

> ⚠️ **Never `@inject` one `@InjectableStore` into another store's constructor.** `@InjectableStore` extracts the class's state via `transformClass()` as soon as the module is imported (decoration time) — well before `AppBootstrap` runs `app.use(pinia)`. Constructor-injecting another store forces tsyringe to instantiate that store's Pinia-backed instance right then, which calls `getActivePinia()` with no active Pinia yet and throws `"getActivePinia() was called but there was no active Pinia"`. Components are fine (they mount after `app.use(pinia)`), and plain services/repositories are fine — only store→store constructor injection is the trap. If a store genuinely needs another store, resolve it lazily at the point of use: `appContainer.resolve(OtherStore)` inside a method, not stored as a constructor-injected field.

**Always use arrow functions.** `EventBus` calls handlers with its own context, so passing `this.methodName` directly will lose `this`. Wrap in an arrow function instead:

```ts
// correct
this.eventBus.registerHandler(WorkoutStartedEvent, (e) => this.onWorkoutStarted(e))

// wrong — this will be undefined inside the handler
this.eventBus.registerHandler(WorkoutStartedEvent, this.onWorkoutStarted)
```

Component example:

```ts
created(): void {
    this.eventBus.registerHandler(WorkoutStartedEvent, async (e) => {
        this.workoutId = e.workoutId
        await this.loadWorkout()
    })
}
```

Store example — register in a one-shot lifecycle method (not `setup()`) and tear down:

```ts
public async start(): Promise<void> {
    const handler = (e: RoutingSyncedEvent) => this.onRoutingSynced(e)
    this.eventBus.registerHandler(RoutingSyncedEvent, handler)
    this.cleanupFns.push(() => this.eventBus.unregisterHandler(RoutingSyncedEvent, handler))
}

public dispose(): void {
    this.cleanupFns.forEach((fn) => fn())
    this.cleanupFns = []
}
```

### Injecting EventBus

```ts
import { EventBus } from '@/services/eventBus/EventBus'
import { inject } from 'tsyringe'

constructor(
    @inject(EventBus) private readonly eventBus: EventBus,
) {
    super()
}
```

## Animations — `motion-v` and `<transition>`

### motion-v

Use `motion-v` for element-level animations (enter, loop, exit). Import `motion` from `'motion-v'` and register specific element variants as named components in `@Component`.

**Registration:**

```ts
import { motion } from 'motion-v'

@Component({
  components: {
    MotionDiv: motion.div,
    MotionSpan: motion.span,
  },
})
```

Use the registered name in `kebab-case` in the template: `<motion-div>`, `<motion-span>`.

**Props:**

| Prop | Purpose |
|---|---|
| `:initial` | State before the element enters |
| `:animate` | Target state (or keyframe array for loops) |
| `:exit` | State when the element leaves (requires `<transition>` wrapper) |
| `:transition` | Timing — `duration`, `delay`, `ease`, `repeat`, `repeatType` |

```vue
<motion-div
    :animate="{ scale: 1, opacity: 1 }"
    :initial="{ scale: 0.5, opacity: 0 }"
    :transition="{ duration: 0.6, ease: 'easeOut' }"
/>

<!-- keyframe loop -->
<motion-div
    :animate="{ opacity: [0.1, 0.8, 0.1] }"
    :initial="{ opacity: 0 }"
    :transition="{ delay: 0.6, duration: 2, repeat: Infinity, repeatType: 'loop' }"
/>
```

### `<transition>` with `:exit`

Wrap a `v-if` / `v-else` block in Vue's `<transition>` when the leaving element needs an exit animation driven by `:exit`. The `<transition>` gives `motion-v` the time to run the exit animation before the element is removed from the DOM.

Use `@after-leave` to run logic (e.g. emit an event) **after** the exit animation fully completes:

```vue
<transition name="splash" @after-leave="$emit('finish')">
  <motion-div
      v-if="loading"
      :exit="{ opacity: 0 }"
      :initial="{ opacity: 1 }"
      :transition="{ duration: 0.5 }"
  >
    <!-- content -->
  </motion-div>
  <div v-else>
    <slot />
  </div>
</transition>
```

- The `name` attribute on `<transition>` is optional but useful for scoped CSS transitions.
- `:exit` only works when the element is inside a `<transition>` — without it the element is removed instantly and the exit animation never runs.
- Do **not** add CSS transition classes (`.splash-leave-active`, etc.) unless you need them alongside `motion-v`; `motion-v` drives the animation itself.
